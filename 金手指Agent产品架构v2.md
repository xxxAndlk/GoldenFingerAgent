# 金手指 Agent 产品架构 v2.0

> **定位**: 玄幻网文「金手指」× 现实 Agent 产品
> **设计原则**: 以**作用范围**划分模块，而非传统功能模块
> **核心命题**: 如何让一个 Agent 像小说主角的金手指一样，带领用户持续成长？

---

## 零、产品隐喻与核心回路

### 0.1 一句话定义

> 金手指是一个**本地优先、持续进化**的个人 Agent 系统。它接收用户的任何问题，自行拆解、规划、执行、校验，并将每一次经验沉淀为可复用的「功法」，越用越强。

### 0.2 核心交互回路

```
用户提出任意问题
       │
       ▼
  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
  │ ① 天机   │───▶│ ② 施法   │───▶│ ③ 验道   │───▶│ ④ 刻碑   │
  │  推演    │    │  执行    │    │  校验    │    │  沉淀    │
  └─────────┘    └─────────┘    └─────────┘    └─────────┘
       │              │              │              │
       └──────────────┴──────────────┴──────────────┘
                      ⑤ 内外结界（贯穿全程）
```

| 网文隐喻 | 产品模块 | 核心职责 |
|----------|----------|----------|
| 天机推演 | 分析规划域 | 拆解问题 → 规划任务 → 组合提示词 → 调用大模型 |
| 施法执行 | 执行调度域 | 解析模型输出 → 调用工具/Skill → 安全沙箱 → 异步编排 |
| 验道校验 | 结果校验域 | 需求结果对比 → 步骤回放验证 → 失败回退 → 自动重构 |
| 刻碑沉淀 | 持久进化域 | 执行总结 → Skill 更新/新建 → 知识沉淀 → 越用越强 |
| 内外结界 | 隔离安全域 | 用户数据永不出境 → 外部数据入境过滤 → 画像持续完善 |

### 0.3 与传统功能模块化的区别

## 一、模块一：天机推演（分析规划域）
### Analysis & Planning Domain

### 1.1 职责边界

> **只做一件事：把用户的任意问题，变成可执行的原子任务 DAG。**

输入：用户原始问题（自然语言 / 语音 / 图片）
输出：`TaskPlan`（包含任务拆解、依赖关系、Skill 匹配、提示词列表）

### 1.2 内部流程

```
用户输入: "我想三个月转行做AI产品经理，帮我规划"
           │
           ▼
  ┌─────────────────────────────┐
  │ 1. 意图分类 + 复杂度评估     │  天机感应
  │    - 简单问答 / 技能学习 /   │
  │      多步任务 / 长期规划     │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 2. 任务拆解引擎             │  推演棋盘
  │    - 将问题拆解为原子任务    │
  │    - 标注任务间依赖关系      │
  │    - 输出 DAG（有向无环图）  │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 3. Skill 匹配引擎           │  功法匹配
  │    - 检索已有 Skill 库      │
  │    - 标记哪些步骤有现成 Skill│
  │    - 标记哪些步骤需要新建    │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 4. 提示词组合引擎           │  传音入密
  │    - 为每个子任务拼装提示词  │
  │    - 注入宿主画像上下文      │
  │    - 注入相关 Skill 知识    │
  │    - 注入安全约束指令       │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 5. 大模型调用编排            │  天机问卦
  │    - 按 DAG 拓扑序发起调用   │
  │    - 并行节点同时发送        │
  │    - 前一节点输出作为后一节点│
  │      输入的上下文            │
  └─────────────┬───────────────┘
                ▼
         输出: TaskPlan → 进入执行域
```

### 1.3 关键设计决策

**Q: 为什么拆解和 LLM 调用都在分析域？**

拆解任务本身可以调用 LLM（元规划），也可以用小模型+规则。最终产出的 `TaskPlan` 是一个完整蓝图，执行域只负责机械执行，不再做决策。**决策与执行分离**是本架构的核心约束。

**Q: 提示词组合引擎为什么放在这里而非执行域？**

提示词的质量直接决定了后续所有步骤的质量。将提示词构建集中在分析域，可以：
- 统一注入用户画像（执行域不应触碰用户画像原始数据）
- 统一注入安全约束（执行域只看到已被「净化」的提示词）
- 方便做提示词的 A/B 测试与版本管理

### 1.4 核心数据结构

```python
from pydantic import BaseModel
from typing import List, Optional, Dict, Any
from enum import Enum

class TaskComplexity(str, Enum):
    SIMPLE_QA     = "simple_qa"       # 一问一答
    SKILL_SINGLE  = "skill_single"     # 调用单个 Skill
    SKILL_CHAIN   = "skill_chain"      # 多个 Skill 串联
    LONG_RUNNING  = "long_running"     # 长期规划，分多阶段

class AtomTask(BaseModel):
    """原子任务：DAG 中的节点"""
    task_id: str
    description: str
    depends_on: List[str]          # 前置任务 ID 列表
    matched_skill: Optional[str]   # 匹配到的已有 Skill 名称
    is_new_skill_needed: bool
    prompt_template: str           # 组装好的提示词模板
    safety_level: int              # 0~3，0=可出境，3=仅本地
    timeout_ms: int

class TaskPlan(BaseModel):
    """任务计划：DAG"""
    plan_id: str
    original_query: str
    complexity: TaskComplexity
    tasks: List[AtomTask]
    execution_order: List[List[str]]  # 拓扑分层，每层可并行
    estimated_duration_ms: int
    host_context_snapshot: Dict[str, Any]  # 本次任务所需的宿主上下文（脱敏版）
```

### 1.5 推荐技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 意图分类 | `outlines` + 微调小模型 | 本地运行，延迟 < 50ms，无需调用外部 LLM |
| 任务拆解 | LangGraph `StateGraph` | 拆解过程本身是图→图的转换，LangGraph 天然表达 |
| Skill 匹配 | ChromaDB 向量检索 | 将任务描述向量化 → 检索最近 Skill |
| 提示词管理 | Jinja2 模板 + YAML | 提示词版本化、可测试、可审计 |
| LLM 调用 | LiteLLM | 统一多模型接口（OpenAI / Claude / 本地模型） |

## 二、模块二：施法执行（执行调度域）
### Execution & Orchestration Domain

### 2.1 职责边界

> **只做一件事：按 TaskPlan 的拓扑序，逐一执行原子任务，管理工具调用，确保安全。**

输入：`TaskPlan`（来自分析域）
输出：`ExecutionReport`（原始结果 + 工具调用日志 + 异常记录）

### 2.2 内部流程

```
TaskPlan 进入执行域
           │
           ▼
  ┌─────────────────────────────┐
  │ 1. 执行编排器 (Orchestrator)│  万法归宗
  │    - 按拓扑分层调度任务      │
  │    - 并行层同时执行          │
  │    - 依赖满足后触发下游      │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 2. 单任务执行循环            │  施法
  │    ┌─────────────────────┐  │
  │    │ a. LLM 推理         │  │  ← 调用大模型，获取回复
  │    │ b. 解析 Tool Call   │  │  ← 从回复中提取 tool_use
  │    │ c. 注入检测         │  │  ← 检查提示词注入风险
  │    │ d. 权限校验         │  │  ← 检查是否有权执行
  │    │ e. 沙箱执行         │  │  ← 在隔离环境中执行工具
  │    │ f. 结果收集         │  │  ← 捕获输出 & 异常
  │    │ g. 循环至 b 或结束  │  │  ← 如果 LLM 需要再调工具
  │    └─────────────────────┘  │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 3. 异步任务管理器            │  分神化念
  │    - 长时间任务后台运行      │
  │    - 进度通知推送            │
  │    - 任务取消/暂停/恢复      │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 4. 异常处理器               │  心魔预警
  │    - LLM 调用失败重试       │
  │    - 工具调用异常降级        │
  │    - 超时熔断               │
  └─────────────┬───────────────┘
                ▼
         输出: ExecutionReport → 进入校验域
```

### 2.3 执行安全纵深防御

这是整个系统**最关键的安全环节**。模型输出是不可信的，工具调用是危险的。

```
                      ┌──────────────┐
                      │  LLM 输出    │
                      └──────┬───────┘
                             │
              ┌──────────────▼──────────────┐
              │  第一层：提示词注入检测       │
              │  - 正则匹配注入签名           │
              │  - LLM-as-Judge 二次审查     │
              │  - 输入/输出长度突变检测      │
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │  第二层：权限校验             │
              │  - 工具白名单检查            │
              │  - 参数范围校验              │
              │  - 调用频率限制 (Rate Limit) │
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │  第三层：沙箱执行             │
              │  - 文件系统：只读挂载 + tmpfs│
              │  - 网络：白名单域名出站       │
              │  - 进程：禁止 fork/exec     │
              │  - 系统调用：Seccomp 过滤    │
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │  第四层：结果校验             │
              │  - 输出格式校验              │
              │  - 数据量异常检测            │
              │  - 敏感信息泄漏扫描          │
              └──────────────────────────────┘
```

### 2.4 核心数据结构

```python
class ExecutionContext(BaseModel):
    """单次执行的完整上下文"""
    task: AtomTask
    messages: List[Dict[str, Any]]      # 完整的 LLM 对话历史
    tool_calls: List[ToolCallLog]       # 工具调用链
    sandbox_id: str                     # 沙箱实例 ID
    started_at: float
    retry_count: int

class ToolCallLog(BaseModel):
    """每次工具调用的审计记录"""
    timestamp: float
    tool_name: str
    params_before_sanitize: Dict   # 模型原始参数（用于注入分析）
    params_after_sanitize: Dict    # 净化后参数（实际使用的）
    result: Any
    error: Optional[str]
    duration_ms: int
    injection_check_passed: bool
    permission_check_passed: bool

class ExecutionReport(BaseModel):
    """执行域输出"""
    execution_id: str
    plan_id: str
    status: str                      # completed / partial_failure / aborted
    task_results: Dict[str, Any]     # task_id → task result
    tool_call_logs: List[ToolCallLog]
    total_duration_ms: int
    anomalies: List[str]             # 执行过程中发现的异常
```

### 2.5 推荐技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 执行编排 | **Temporal.io** (Python SDK) | 工业级工作流引擎，原生支持重试/超时/补偿/可视化管理 |
| 沙箱执行 | **Docker** + **gVisor** | 每个原子任务跑在独立容器中，gVisor 提供内核级隔离 |
| 注入检测 | **LLM Guard** + 自定义规则 | 专为 LLM 输入/输出安全设计的库，支持注入/越狱检测 |
| 权限控制 | **OPA (Open Policy Agent)** | 声明式策略引擎，权限规则与代码解耦 |
| 异步管理 | Temporal 内置 + **Celery** | Temporal 处理工作流级异步，Celery 处理工具级异步 |
| 模型调用 | **LiteLLM** | 统一接口，内置重试/降级/速率限制 |

## 三、模块三：验道校验（结果校验域）
### Verification & Correction Domain

### 3.1 职责边界

> **只做一件事：验证执行结果是否符合原始需求，不符合则回退或自动重构。**

输入：`ExecutionReport` + 原始需求
输出：`VerificationReport`（通过/不通过 + 修正后的结果或回退指令）

### 3.2 校验三层模型

```
                     ┌──────────────────────────┐
                     │  原始需求:                 │
                     │  "帮我生成一份竞品分析报告" │
                     └────────────┬─────────────┘
                                  │
            ┌─────────────────────▼─────────────────────┐
            │          第一层：结构校验 (Structure)       │
            │  - 文件是否生成？格式是否正确？             │
            │  - 必要章节是否齐全？                       │
            │  - 数据字段是否有缺失？                     │
            │  校验方式：Schema 匹配 + 规则引擎           │
            └─────────────────────┬─────────────────────┘
                                  │
            ┌─────────────────────▼─────────────────────┐
            │          第二层：内容校验 (Content)         │
            │  - 内容是否与需求一致？                      │
            │  - 是否有事实性错误？                       │
            │  - 是否遗漏关键信息？                       │
            │  校验方式：LLM-as-Judge + RAG 事实核查      │
            └─────────────────────┬─────────────────────┘
                                  │
            ┌─────────────────────▼─────────────────────┐
            │          第三层：步骤回放 (Replay)          │
            │  - 关键步骤是否真的执行成功？               │
            │  - 数据写入是否正确？                       │
            │  - API 调用是否有副作用残留？               │
            │  校验方式：函数式回放 + 幂等性检查          │
            └─────────────────────┬─────────────────────┘
                                  │
                          ┌───────▼───────┐
                          │  通过 / 不通过 │
                          └───────┬───────┘
                                  │
                 ┌────────────────┼────────────────┐
                 ▼                ▼                ▼
           ┌──────────┐   ┌──────────┐   ┌──────────┐
           │ 全部通过  │   │ 可修复   │   │ 不可修复 │
           │ → 持久域  │   │ → 重构   │   │ → 回退   │
           └──────────┘   └──────────┘   └──────────┘
```

### 3.3 回退与重构机制

```
校验失败分支决策树：

校验失败
  │
  ├── 失败范围 ≤ 单任务？
  │     ├── 是 → 重试该任务（最多 N 次）
  │     │        └── 仍失败 → 标记为「可降级」？
  │     │              ├── 是 → 跳过该任务，记录缺口
  │     │              └── 否 → 回退整条链路
  │     │
  │     └── 否 → 失败涉及多任务？
  │           ├── 有明确根因 → 修复根因任务 → 重放受影响下游
  │           └── 无明确根因 → 重构 TaskPlan → 重新进入执行域
  │
  └── 用户需求本身有歧义？
        └── 是 → 向用户请求澄清 → 重新进入天机推演
```

### 3.4 核心数据结构

```python
class VerificationResult(BaseModel):
    check_name: str
    passed: bool
    detail: str
    evidence: Optional[str]         # 校验证据（截图路径/日志片段/API返回值）
    fix_suggestion: Optional[str]   # 修复建议

class VerificationReport(BaseModel):
    verification_id: str
    execution_id: str
    original_requirement: str       # 原始需求一句话
    overall_pass: bool
    structure_checks: List[VerificationResult]
    content_checks: List[VerificationResult]
    replay_checks: List[VerificationResult]
    action: str                     # "pass" | "retry" | "rollback" | "reconstruct" | "ask_user"

class RollbackPlan(BaseModel):
    """回退计划"""
    affected_tasks: List[str]
    rollback_actions: List[Dict]    # 每一步回退操作
    side_effects_to_clean: List[str] # 需要清理的副作用
```

### 3.5 推荐技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| Schema 校验 | **Pydantic** + **JSON Schema** | 声明式结构校验，与 Python 生态无缝集成 |
| 内容校验 | **LangChain Criteria Evaluator** + 自建 RAG | LLM 评估输出质量，RAG 做事实核查 |
| 步骤回放 | **Temporal Replay** + 自定义 Assertion | Temporal 原生支持 workflow replay |
| 数据写入校验 | SQLite `PRAGMA integrity_check` | 验证本地数据库写入正确性 |
| 结果存储 | 执行中间结果写入 **SQLite**，最终产物写入文件系统 | 结构数据用 DB，非结构数据用文件 |

## 四、模块四：刻碑沉淀（持久进化域）
### Persistence & Evolution Domain

### 4.1 职责边界

> **只做一件事：将本次执行的经验，转化为可复用的 Skill 知识，让系统越用越强。**

这是「金手指」与普通 Agent 的**本质区别**——普通 Agent 每次都是新鲜的，金手指会成长。

### 4.2 沉淀五道工序

```
ExecutionReport + VerificationReport 进入持久域
           │
           ▼
  ┌─────────────────────────────┐
  │ 1. 执行摘要生成              │  著书立说
  │    - LLM 将完整执行过程     │
  │      压缩为结构化摘要        │
  │    - 提取: 做了什么→怎么做的│
  │      →遇到什么问题→怎么解决 │
  └─────────────┬───────────────┘
                ▼
  ┌─────────────────────────────┐
  │ 2. Skill 缺口分析            │  寻道补缺
  │    - 本次是否有 Skill 没有   │
```

### 4.3 Skill 自我进化机制

```python
class SkillEvolutionEngine:
    """
    Skill 进化引擎：Skill 不是静态代码，而是持续更新的知识单元

    每个 Skill 包含：
    - manifest:   声明信息（名称/分类/境界要求）
    - knowledge:  知识条目集合（可增删改）
    - templates:  提示词模板（可版本迭代）
    - tools:      关联工具列表（可扩充）
    - stats:      使用统计（成功率/平均耗时/用户满意度）
    """

    def analyze_gap(self, task_plan: TaskPlan,
                    execution_report: ExecutionReport) -> GapReport:
        """
        分析本次执行的 Skill 缺口
        判断依据：
        - 有步骤未匹配到任何 Skill
        - 已匹配 Skill 但执行中有多次重试
        - 已验证通过的新执行模式
        """
        gaps = []

        for task in task_plan.tasks:
            if task.is_new_skill_needed:
                gaps.append(Gap(
                    type="missing_skill",
                    task_description=task.description,
                    suggested_skill_name=self._infer_skill_name(task)
                ))

        for log in execution_report.tool_call_logs:
            if log.retry_count > 2:
                gaps.append(Gap(
                    type="skill_deficiency",
                    skill_name=task.matched_skill,
                    deficiency=log.error,
                    suggestion=self._suggest_improvement(log)
                ))

        return GapReport(gaps=gaps)

    def update_skill(self, skill_name: str,
                     execution_summary: ExecutionSummary) -> Skill:
        """
        更新已有 Skill 的知识
        - 追加成功案例到 knowledge base
        - 调整 prompt 模板中的边缘情况处理
        - 更新工具调用的最佳参数
        """
        skill = self._load_skill(skill_name)

        # 追加知识条目
        new_entry = KnowledgeEntry(
            source_execution=execution_summary.execution_id,
            context=execution_summary.context,
            pattern=execution_summary.success_pattern,
            pitfalls=execution_summary.pitfalls,
            created_at=now()
        )
        skill.knowledge.append(new_entry)

        # 更新提示词模板（如果发现了更有效的表达方式）
        if execution_summary.prompt_improvement:
            skill.templates.update(
                version=skill.templates.version + 1,
                content=execution_summary.prompt_improvement
            )

        # 更新统计
        skill.stats.total_uses += 1
        skill.stats.success_rate = self._recalculate_success_rate(skill)

        return skill

    def create_skill(self, gap: Gap,
                     execution_summary: ExecutionSummary) -> Skill:
        """
        从执行经验中创建新 Skill
        """
        manifest = SkillManifest(
            name=self._generate_skill_name(gap),
            display_name=gap.suggested_display_name,
            category=self._infer_category(gap),
            realm_requirement=self._infer_realm(gap),
            version="0.1.0",
            created_from=execution_summary.execution_id
        )

        # 从本次执行中提取初始知识
        seed_knowledge = [
            KnowledgeEntry(
                source_execution=execution_summary.execution_id,
                context=execution_summary.context,
                pattern=execution_summary.success_pattern,
                pitfalls=execution_summary.pitfalls
            )
        ]

        # 生成初始提示词模板
        initial_template = self._generate_prompt_template(
            gap.task_description,
            execution_summary
        )

        return Skill(
            manifest=manifest,
            knowledge=seed_knowledge,
            templates=initial_template,
            tools=execution_summary.tools_used,
            stats=SkillStats(total_uses=1)
        )
```

### 4.4 推荐技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| Skill 存储 | **ChromaDB** (向量) + **SQLite** (结构化) | Skill 知识需要向量检索 + 结构化元数据 |
| 宿主画像 | **SQLite** 本地加密 (SQLCipher) | 画像数据完全本地，加密存储 |
| 知识图谱 | **NetworkX** + 本地 JSON | 轻量，个人知识图谱不需要图数据库 |
| 摘要生成 | LLM (结构化输出) | 用 `outlines` 约束输出格式保证结构化 |
| 版本管理 | **git** (Skill 定义文件) | Skill 是代码+知识的混合体，用 git 做版本管理最自然 |

## 七、与 v1.0 设计文档的关键差异

> **设计总结**
>
> v2.0 的核心思想是**让 Agent 像小说金手指一样服务用户**：
> - **天机推演** → 一个问题进来，自动拆解为可执行计划
> - **施法执行** → 调度工具与模型，安全落地
> - **验道校验** → 检查结果对不对，不对就修正
> - **刻碑沉淀** → 把这次的经验存下来，下次更强
> - **内外结界** → 全程保护宿主数据安全
>
> 五个域形成闭环，每一圈循环都比上一圈更强——这就是「修炼」。
